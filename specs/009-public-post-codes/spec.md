# Feature Specification: Public Post Codes

**Feature Branch**: `main`
**Created**: 2026-07-22
**Status**: Ready for planning
**Input**: Replace obvious sequential post IDs in public blog URLs with stable opaque codes while retaining numeric IDs for internal APIs and command-line use.

## User Scenarios & Testing

### User Story 1 - Share a non-sequential article URL (Priority: P1)

As a reader or author, I want public article links to use an opaque post code so that the URL does not advertise the post's sequential database ID.

**Why this priority**: Public links are visible in the address bar, search results, social shares, feeds, and copied references.

**Independent Test**: Open any published article from a public listing and confirm that the resulting article URL contains a stable opaque code rather than its decimal post ID.

**Acceptance Scenarios**:

1. **Given** a published article appears on a public listing, **When** a reader follows it, **Then** the article URL uses the article's opaque public code.
2. **Given** the same article is linked from different public surfaces, **When** those links are inspected, **Then** every surface uses the same public code.
3. **Given** a reader opens a coded article URL directly or refreshes it, **When** the page loads, **Then** the correct published article appears.
4. **Given** an article URL is represented in search metadata, a feed, a sitemap, or a share action, **When** that URL is generated, **Then** it uses the same public code as browser navigation.

### User Story 2 - Preserve existing article links (Priority: P1)

As a reader following an older numeric article link, I want it to continue working and lead to the new coded URL so that bookmarks and external references do not break.

**Why this priority**: Existing numeric URLs are already indexed and shared publicly; breaking them would create avoidable link rot.

**Independent Test**: Open a known numeric article URL and confirm it resolves to the same article at its coded canonical URL.

**Acceptance Scenarios**:

1. **Given** a published article has an existing numeric URL, **When** a reader opens that URL, **Then** the reader is redirected to the coded URL for the same article.
2. **Given** a crawler requests an existing numeric article URL, **When** the response is produced, **Then** the coded URL is identified as the permanent canonical destination.
3. **Given** a malformed or unknown public code, **When** a reader opens it, **Then** no unrelated article is displayed.

### User Story 3 - Keep internal identifiers operational (Priority: P2)

As an operator, I want internal post requests and diagnostics to continue using numeric post IDs so that existing command-line and backend workflows remain unchanged.

**Why this priority**: The public URL change is cosmetic and must not expand into a backend data migration or API contract change.

**Independent Test**: Use the existing numeric post API for a known article and confirm that its request and response behavior is unchanged.

**Acceptance Scenarios**:

1. **Given** an operator knows a post's numeric ID, **When** the existing internal request is made, **Then** it behaves exactly as before.
2. **Given** public navigation uses a code, **When** the application retrieves article data, **Then** it resolves the code without changing the numeric backend contract.

## Requirements

### Functional Requirements

- **FR-001**: Every public article URL MUST use one stable opaque code derived from the article's numeric ID.
- **FR-002**: The public code MUST NOT contain the decimal ID as an obvious substring or suffix.
- **FR-003**: The same article MUST receive the same code across browser navigation, canonical metadata, structured data, feeds, sitemaps, and share targets.
- **FR-004**: A coded article URL MUST resolve to exactly one numeric post ID.
- **FR-005**: Invalid, malformed, or non-canonical codes MUST NOT resolve to an unrelated post.
- **FR-006**: Existing numeric article URLs MUST remain supported and permanently redirect to the corresponding coded URL.
- **FR-007**: Existing numeric backend post APIs MUST remain unchanged.
- **FR-008**: Protected profile and analytics routes MUST retain their existing identifier behavior.
- **FR-009**: The coding configuration MUST remain fixed so previously issued public URLs remain valid.
- **FR-010**: The feature MUST be described as identifier obfuscation, not as an authorization or security boundary.

## Success Criteria

- **SC-001**: 100% of public article links generated by tested reader-facing surfaces use opaque codes rather than decimal IDs.
- **SC-002**: 100% of tested coded URLs load the same article as their corresponding numeric IDs.
- **SC-003**: 100% of tested legacy numeric article URLs identify the coded URL as their permanent destination.
- **SC-004**: Invalid and non-canonical codes resolve to no article in all tested cases.
- **SC-005**: Existing numeric internal post requests exhibit no contract changes.
- **SC-006**: Focused route, metadata, feed, sitemap, and frontend navigation regressions pass before handoff.

## Assumptions

- The goal is to make sequential IDs non-obvious in public URLs, not to prevent determined enumeration or replace authorization.
- Public codes are deterministic and reversible so no database migration or additional stored identifier is required.
- Public codes are permanent once introduced; their configuration is part of the public-link contract.
- Numeric IDs may remain visible in network responses and internal tooling.
- Existing profile and analytics URLs are outside this public-reader URL change.
